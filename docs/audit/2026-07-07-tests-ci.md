## CI gate map (`.github/workflows/`)
`ci.yml` runs on every PR + push to main â€” 4 parallel jobs, with `concurrency: cancel-in-progress`:
1. `lint-test` â€” ruff + `pytest -q` (Python)
2. `frontend` â€” npm build + svelte-check + vitest unit
3. `worker` â€” CF Worker vitest (proxy.js)
4. `ui-smoke` â€” boots the real FastAPI app serving the built SPA, Playwright-drives it â€” the pre-merge stand-in for the dispatch-only cloud e2e

**BUT only `lint-test` (+ GitGuardian) is a required status check.** Branch protection on `main` (queried via `gh api .../branches/main/protection`, 2026-07-07: `strict: false`, `enforce_admins: false`) does NOT require `frontend`, `worker`, or `ui-smoke` â€” they run on every PR, but a red run on any of them does not block merge. So the frontend is *exercised* on every PR but **not** actually merge-gated â€” see gap #1. Other workflows:
- `e2e.yml` â€” **workflow_dispatch only**; never auto on PR/push. It's the only thing exercising the real deployed coordinator + Firestore + a real GitHub PR lifecycle. The live-system path is entirely manual â€” nothing auto-catches a live regression.
- `iac.yml` â€” runs on **every** PR (deliberately no `paths:` filter â€” the workflow's own comment explains why): static HCL gate + `tofu fmt`/`validate` (no live GCP creds, `pull_request` only, never `pull_request_target`). Not a required check either. A separate `workflow_dispatch`-only `plan-builder` job (main-only, WIF-gated) does the real `tofu plan` â€” also manual.
- `demo-reset.yml` cron â€” `service-reset` every 2h (payment-demo drift pre-read+diff, autonomy/pause safety-net) + daily 6:00 JST `lodash-repin`. **Failure notification = GitHub's built-in "email the repo owner on a failed scheduled run", nothing else** (no Slack/webhook). A degraded read intentionally fails the job at the summary step so that email fires. If nobody watches email, a broken demo-reset is silent to judges until the demo visibly breaks.

## Strengths
- **Prompt byte-golden pinning is deliberate and layered, not lazy.** `test_drift_workload_loads.py` pins full LLM prompt text verbatim (framed as an audit mechanism: "the diff in code review is exactly the prompt edit") PLUS separate `pins_*_rule` tests asserting specific safety phrases independently, so an intentional golden bump can't silently drop a rule (e.g. "NEVER invent a revision name").
- **`InfraDiagram.test.ts` (1726 lines, 70 tests) is genuinely behavioral** â€” tests like "refresh coalescing + last-applied-wins (livelock regression)" and "an in-flight overlay fetch resolving AFTER exit never lands" assert real async/race invariants, plus regression tests tied to past prod incidents.
- **`test_iac_approval_get.py`** tests hard invariants over shape: it asserts the read-only GET "never mints a plan approval, never calls the tofu-apply worker, never reads `plan_approvals`" â€” testing what must NOT happen.
- `iac.yml` and `demo-reset.yml` carry unusually thorough inline security rationale (WIF scoping, TOCTOU closure, actions-injection prevention via `env:` indirection) â€” CI written with an adversarial mindset.
