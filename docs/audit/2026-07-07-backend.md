# DriftScribe backend audit — 2026-07-07

Scope: `agent/` (coordinator), `workers/*`, `driftscribe_lib/`, the CF Worker proxy. Read-only. Covered `agent/main.py` (6,293 lines), `trace_fetcher.py`, `infra_graph_cache_store.py`, `per_pr_cache.py`, `state_store.py`, `config.py`, `auth.py`, `driftscribe_lib/github.py`, `cf_access.py`, `fanout.py`, `worker_client.py`, all `workers/*/main.py`, `tofu_runner.py`.

This is a mature, heavily Codex-reviewed codebase — most obvious failure modes (unbounded caches, missing timeouts, CSRF, blocking calls) were already deliberately handled. The findings below are what slipped through. Findings 1 and 2 were spot-checked against the code and **confirmed**.

## Findings (ranked)

### 1. HIGH — blocking sync network call inside an async route stalls the whole event loop
`agent/main.py:1912`, inside `async def eventarc` (`main.py:1804`): `verify_oauth2_token(token, _GOOGLE_AUTH_TRANSPORT, audience=...)` is called directly in the coroutine body — not via `Depends()` (which FastAPI thread-pools) and not wrapped in `asyncio.to_thread`. It synchronously fetches Google's JWKS certs over HTTPS via the `requests`-based transport. **The code's own comment (main.py:1904-1907) confirms this transport does NOT cache JWKS across calls** — so every `/eventarc` call pays a real blocking network round-trip on the coroutine's thread. uvicorn shares one event loop across all concurrent async requests (including every live `/chat` SSE stream), so this stalls unrelated in-flight requests for the fetch duration — worst case up to google-auth's default transport timeout (~2 min) if Google's endpoint is slow/unreachable. `/eventarc` is the live wiring for the autonomous drift trigger (fires on Cloud Run audit-log events for payment-demo), so this is live traffic.

**Verified:** `eventarc` is `async def` (main.py:1804); the call at 1912 is a bare synchronous call in the coroutine.

**Fix:** `claims = await asyncio.to_thread(verify_oauth2_token, token, _GOOGLE_AUTH_TRANSPORT, audience=s.eventarc_audience)`.

### 2. MEDIUM — unbounded in-process cache, inconsistent with its own sibling
`agent/main.py:631` — `_TRACE_CACHE: dict[str, tuple[float, dict]] = {}` (backs `/trace/{trace_id}`) has **no size cap**; entries are only removed lazily on read when TTL-expired (`_cache_get`, main.py:738-756). A trace that completes and is never re-opened (the common case for one-shot recheck/chat runs) sits in memory forever. This is the exact leak the code diagnosed and fixed for the neighboring `_TRACE_OBSERVATIONS` dict three lines below (`_OBSERVATIONS_SOFT_CAP = 1024` at main.py:654 with FIFO eviction at 728-731; its comment literally says "a trace polled once... but never polled again leaves an observation entry forever... unbounded slow leak") — **the same fix was never applied to `_TRACE_CACHE`.** With the public demo window open continuously through the Aug 19 finals, plus autonomous rechecks and the 2-hourly demo-reset cron generating a steady stream of distinct trace_ids, this grows for the life of each instance.

**Verified:** `_TRACE_CACHE` (631) has `_TRACE_CACHE_TTL_S` but no cap; `_TRACE_OBSERVATIONS` (643) has the soft-cap + FIFO eviction.

**Fix:** apply the same FIFO soft-cap pattern already used for `_TRACE_OBSERVATIONS`.

### 3. MEDIUM — external GitHub Advisory API call has no error handling, unlike its sibling worker's degradation contract
`workers/upgrade_reader/main.py:195-201`, `_lookup_advisories()` does `requests.get(...)` + `resp.raise_for_status()` with no try/except, called once per dependency inside `read()` (main.py:248-297, no surrounding try/except either). A transient GitHub Advisory API failure (rate limit, 5xx, network blip) raises unhandled → bare FastAPI 500 → `agent/worker_client.py`'s `call()` maps any non-2xx to `WorkerClientError` → the whole `/chat` turn for the upgrade/Patch crew fails with a 502. Contrast `workers/infra_reader/main.py`'s explicit pattern: "a CAI permission/availability failure soft-fails to an HTTP 200... lets chat narrate the partial degradation." The advisory lookup should degrade the same way (return the dependency with `advisories:[]` + an error flag) instead of failing the whole turn.

### 4. LOW / worth watching — full unbounded Firestore collection scan on every decisions read
`agent/state_store.py:539` (`list_decisions`) does `list(self._decisions.stream())` — fetches every doc in the decisions collection, sorts + trims client-side. (`list_decisions_for_pr` at `:568` is **not** a full scan — it filters server-side with `.where("pr_number", ==)` and fetches only that PR's handful of docs; an earlier draft of this finding lumped it in wrongly.) This is a **documented, deliberate tradeoff** (comment at state_store.py:528-532: "hackathon decision volume is in the hundreds, not millions... if this scales past that, swap to a server-side ordered query"), not a bug — but the assumption is under more pressure with the public window open continuously through finals. `GET /decisions` and the `read_team_log` tool both pay this cost every call. Worth a volume check before finals.

## Done well (don't touch)
- **`trace_fetcher.py` + the `/trace` Future-based timeout boundary** (main.py:611-625, 3376-3396): a genuinely well-engineered fix for a real prior outage — two-phase narrow→wide Cloud Logging query with measured latency numbers in comments, plus a real `concurrent.futures.Future.result(timeout=...)` wall clock around a client library with no native timeout. The strongest postmortem-driven engineering in the codebase.
- **`state_store.py` Firestore transactions:** `evict_cached_decision` (CAS delete) and `append_turns` (atomic seq allocation + doc creation) correctly use `@firestore.transactional` to close real concurrent-writer races, with comments explaining which race each closes.
- **Worker isolation model:** every `workers/*/main.py` documents its blast radius in a consistent "Layer 1 IAM / Layer 2 payload-intent / Layer 3 inter-service auth" structure, backed by closed pydantic schemas (`extra="forbid"`) and deploy-time repo/path allowlists — applied uniformly across all 9 workers.

## Not found (flagged as risk areas, checked, clean)
The classic race-condition/state-corruption bugs (event-key double-claims, decision-claim races) were **not** present — decision claiming, conversation turn append, and PR merge-then-approve all have explicit CAS/transaction guards with comments citing the specific race closed.
