DriftScribe — Multi-Agent Architecture

Workload-aware coordinator + six execute-only workers on Cloud Run, with Google's Developer Knowledge MCP attached at the coordinator only. Two workloads ship today: drift (Cloud Run env vs ops contract) and upgrade (npm package.json vs GitHub Advisory DB). Layer-0 tool registry is workload-scoped via enabled_tool_names; per-worker hardcoded policies enforce Layer 2; HITL gates the drift rollback path. See docs/architecture/multi-agent-design.md for the full prose.

1. Multi-workload trigger fan-in

Three trigger sources fan in to a workload routing layer (Drift / Upgrade), which selects the workload-scoped agent inside the Coordinator. Each workload sees only its own per-workload tool inventory (Layer 0 capability bound). The Developer Knowledge MCP attaches at the Coordinator only — workers never reason and never query docs. The HITL diamond gates the Coordinator→Rollback edge (drift only); rollback /execute only fires after an operator approves at /approvals/{id}. Eventarc hardcodes workload=drift server-side; /chat routes by the request's workload field (drift or upgrade); /recheck serves drift only in this build (workload=upgrade returns 503 — a workload-specific /recheck pipeline for upgrade is post-Phase-17).

2. Layered safety boundary

For each service, an outer dashed box (Layer 1: IAM scope of its deploy-time service account) wraps an inner solid box (Layer 2: hardcoded payload-intent policy). The Coordinator's inner box is intentionally empty — it has no mutate caps on payment-demo — and carries the Phase 17 Layer 0 note that Developer Knowledge MCP tools attach only at the coordinator, scoped per workload via enabled_tool_names. The HITL diamond intercepts the Coordinator→Rollback flow. The two upgrade workers carry an env-pinned UPGRADE_TARGET_REPO + lockfile path regex; upgrade-docs also runs a deterministic post-LLM validator before any GitHub call. Every worker row shows a small no MCP badge to reinforce that reasoning (and therefore MCP access) is coordinator-only.

Reading the rows: Layer 1 (left) is what the service account could do at the Google Cloud IAM layer; Layer 2 (right) is what the worker code will do regardless of request shape. Both must be breached to mutate payment-demo or open an upgrade PR. The drift Rollback path additionally requires a one-shot HMAC token minted by operator action, and the Upgrade Docs path additionally runs the post-LLM deterministic validator before any GitHub call — so a prompt-injected LLM still cannot smuggle a major-version bump, a downgrade, an off-allowlist lockfile path, or a non-GHSA advisory URL. Workload-scoped IAM means the coordinator's run.invoker on the drift workers does not extend to the upgrade workers, and vice versa.
