DriftScribe — Multi-Agent Architecture

Workload-aware coordinator + eight execute-only workers on Cloud Run, with Google's Developer Knowledge MCP attached at the coordinator only. Four workloads ship today: drift (Cloud Run env vs ops contract), upgrade (npm package.json vs GitHub Advisory DB), explore (strictly read-only investigation), and provision (agent-authored OpenTofu PRs). Layer-0 tool registry is workload-scoped via enabled_tool_names; per-worker hardcoded policies enforce Layer 2; HITL gates the drift rollback and the downstream infra apply. The sole infra mutator, tofu-apply, runs downstream of a plan-bound approval and is not coordinator-invokable. See docs/architecture/multi-agent-design.md for the full prose.

1. Multi-workload trigger fan-in & gated apply

Three trigger sources fan in to a four-way workload router (drift / upgrade / explore / provision), selecting the workload-scoped agent inside the Coordinator. Each workload sees only its own Layer-0 tool slice — the two chat-only workloads make this vivid: explore carries zero mutation tools, provision exactly one (provision_open_infra_pr → opens an iac/ PR, never applies). The Developer Knowledge MCP attaches at the Coordinator only. Eventarc hardcodes workload=drift; /chat routes by the request's workload field (any of the four); /recheck serves drift only (upgrade → 503; explore/provision are chat-only). HITL diamonds gate the drift rollback and — downstream, outside the coordinator's reach — the tofu-apply infra apply behind /iac-approvals/{pr}.

2. Layered safety boundary

For each service, an outer dashed box (Layer 1: IAM scope of its deploy-time service account) wraps an inner solid box (Layer 2: hardcoded payload-intent policy). The Coordinator's inner box is intentionally empty — it has no mutate caps on payment-demo — and notes that Developer Knowledge MCP tools attach only at the coordinator. Rows 7–8 add the infra workers: infra-reader (strictly read-only Cloud Asset Inventory metadata) and tofu-editor (an iac/-only PR author behind the AGENT static gate, holding no GCP role). The final row, tofu-apply, sits below a divider outside the coordinator's reach: it is the sole infra mutator and applies only behind a plan-bound HMAC approval. HITL diamonds gate the drift rollback (left gutter) and the infra apply. Every worker row carries a no MCP badge.

Reading the rows: Layer 1 (left) is what the service account could do at the Google Cloud IAM layer; Layer 2 (right) is what the worker code will do regardless of request shape. Both must be breached to mutate payment-demo, open a PR, or apply infra. The drift Rollback path additionally requires a one-shot HMAC token minted by operator action; the Upgrade Docs path runs the post-LLM deterministic validator before any GitHub call; the tofu-editor re-runs the AGENT static gate (which bans secret literals, control-plane, and IAM edits) before any write; and the downstream tofu-apply — the sole infra mutator, which the coordinator cannot invoke — verifies a plan-bound HMAC approval and applies only the trusted plan triplet. So a prompt-injected LLM still cannot smuggle a major-version bump, a downgrade, an off-allowlist path, a secret-baring IaC change, or an unapproved apply. Workload-scoped IAM means the coordinator's run.invoker on one workload's workers does not extend to another's.
